1.1 — LLMs vs ChatModels

⚠ LLM (ancienne API)
  • Prend une string en entrée
  • Retourne une string
  • API "completion" (GPT-2 style)
  • Moins expressif pour les rôles
✅ ChatModel (recommandé)
  • Prend une liste de messages
  • Retourne un AIMessage
  • System / Human / AI séparés
  • Standard moderne — utilisez toujours celui-ci

1.3 — LCEL : LangChain Expression Language ⭐

🎯

Pareto 80/20 — Concept le plus important

LCEL est le cœur de LangChain moderne. L'opérateur | (pipe) compose des Runnables en séquence. Tout dans LangChain implémente l'interface Runnable : invoke, ainvoke, stream, astream, batch, abatch.

module_01/03_lcel.py
python 
from langchain_ollama import ChatOllama
```

from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import (
RunnablePassthrough,
RunnableLambda,
RunnableParallel,
RunnableBranch,
)

llm = ChatOllama(model=“llama3.2”)
prompt = ChatPromptTemplate.from_messages([
(“system”, “Tu es un assistant concis.”),
(“human”, ”{question}”),
])

# ─── 1. Chain simple : le pipe | ────────────────────────────────
chain = prompt | llm | StrOutputParser()

# invoke — synchrone
result = chain.invoke({“question”: “Qu’est-ce que Python?”})

# stream — chunk par chunk (UX améliorée)
for chunk in chain.stream({“question”: “Explique le ML en 3 points”}):
print(chunk, end=””, flush=True)

# batch — traitement parallèle automatique
results = chain.batch([
{“question”: “Python?”},
{“question”: “JavaScript?”},
{“question”: “Rust?”},
])  # Exécutés en parallèle — bien plus rapide que 3 invokes!

# ainvoke — asynchrone (pour FastAPI, etc.)
import asyncio
result = asyncio.run(chain.ainvoke({“question”: “Test async”}))

# ─── 2. RunnablePassthrough ─────────────────────────────────────
# Passe la valeur sans la modifier — essentiel pour le RAG
chain_enriched = (
{
“context”: RunnableLambda(lambda x: f”Contexte: {x[‘topic’]}”),
“question”: RunnablePassthrough(),  # question inchangée
}
| ChatPromptTemplate.from_messages([
(“human”, ”{context}\n\nQuestion: {question}”)
])
| llm | StrOutputParser()
)

# ─── 3. RunnableParallel — exécution simultanée ─────────────────
parallel = RunnableParallel({
“resume”:   prompt | llm | StrOutputParser(),
“question”: RunnablePassthrough(),  # garder la question originale
})
result = parallel.invoke({“question”: “Explique les LLMs”})
# result = {“resume”: “…”, “question”: {“question”: “…”}}

# ─── 4. RunnableBranch — branchement conditionnel ──────────────
branch = RunnableBranch(
(lambda x: “code” in x[“question”].lower(), chain),  # si “code” → chain
chain,  # default
)

# ─── 5. Utilitaires de robustesse ───────────────────────────────
chain_safe = chain.with_retry(stop_after_attempt=3)   # retry auto
chain_fb   = chain.with_fallbacks([ChatOllama(model=“mistral”) | StrOutputParser()])
chain_cfg  = chain.with_config({“run_name”: “mon_agent”})  # pour LangSmith
```

Question 01 / QCM
Quelle est la différence fondamentale entre LLM et ChatModel dans LangChain ?
ALLM est plus performant que ChatModel
BLLM prend une string, ChatModel prend des messages structurés
CChatModel ne supporte que les modèles OpenAI
DIl n'y a aucune différence pratique
✓ Réponse B. LLM : string → string (ancienne API completion). ChatModel : List[BaseMessage] → AIMessage. Préférez toujours ChatModel pour les nouveaux projets car il supporte les rôles System/Human/AI et est le standard moderne.
Question 02 / QCM
Que fait chain.batch([input1, input2, input3]) comparé à 3 appels invoke séparés ?
AStrictement identique
BExécute les 3 en parallèle — plus rapide
CLes combine en un seul appel LLM
DN'est utile que pour les modèles OpenAI
✓ Réponse B. batch() utilise le threading pour exécuter les invocations en parallèle. C'est bien plus efficace pour traiter de nombreuses requêtes. Le paramètre max_concurrency permet de contrôler le degré de parallélisme.
Question 03 / Ouverte
Expliquez pourquoi RunnablePassthrough est essentiel dans une chain RAG et donnez un exemple concret.
✓ Explication. Dans un RAG, on a besoin de deux informations simultanément pour le prompt final : (1) les documents récupérés et (2) la question originale. RunnablePassthrough permet de "brancher" la question sans la modifier, pendant que le retriever traite sa propre branche.

Exemple : {"context": retriever | format_docs, "question": RunnablePassthrough()} — la question arrive dans les deux branches mais seule celle de droite reste intacte.

Pourquoi la mémoire est indispensable

Les LLMs sont stateless par nature : chaque appel est totalement indépendant. Sans mémoire, l'assistant "oublie" tout à chaque message. La mémoire simule la continuité conversationnelle en injectant l'historique dans chaque nouveau prompt.

3.1 — Mémoire conversationnelle moderne (LCEL)

module_03/01_memory_lcel.py
python 
from langchain_ollama import ChatOllama
```

from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.chat_history import InMemoryChatMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory
from langchain_core.output_parsers import StrOutputParser

llm = ChatOllama(model=“llama3.2”)

# ─── Template avec placeholder pour l’historique ────────────────
prompt = ChatPromptTemplate.from_messages([
(“system”, ””“Tu es un assistant IA serviable.
Tu utilises l’historique de conversation pour répondre de façon cohérente.
Rappelle-toi des informations partagées précédemment.”””),
MessagesPlaceholder(variable_name=“history”),  # historique inséré ici
(“human”, ”{input}”),
])

chain = prompt | llm | StrOutputParser()

# ─── Store multi-sessions (dict par session_id) ─────────────────
store: dict[str, InMemoryChatMessageHistory] = {}

def get_session_history(session_id: str) -> InMemoryChatMessageHistory:
””“Retourne ou crée l’historique pour une session.”””
if session_id not in store:
store[session_id] = InMemoryChatMessageHistory()
return store[session_id]

# ─── Chain avec historique automatique ──────────────────────────
chain_with_history = RunnableWithMessageHistory(
chain,
get_session_history,
input_messages_key=“input”,
history_messages_key=“history”,
)

# ─── Utilisation — même config = même session ────────────────────
config_alice = {“configurable”: {“session_id”: “alice”}}
config_bob   = {“configurable”: {“session_id”: “bob”}}

# Session Alice
r1 = chain_with_history.invoke({“input”: “Je m’appelle Alice et j’adore Python.”}, config_alice)
r2 = chain_with_history.invoke({“input”: “Quel est mon langage préféré?”}, config_alice)
print(r2)  # “Tu aimes Python, Alice!”

# Session Bob — indépendante
r3 = chain_with_history.invoke({“input”: “Je m’appelle Bob.”}, config_bob)
r4 = chain_with_history.invoke({“input”: “Quel est mon prénom?”}, config_bob)
print(r4)  # “Tu t’appelles Bob.” — sans interférence avec Alice

Question 01 / QCM
Dans une architecture RAG, quel est le rôle exact du chunk_overlap dans le TextSplitter ?
ARéduire la taille des embeddings
BÉviter de perdre le sens aux jonctions entre chunks adjacents
CAugmenter la vitesse d'indexation
DFiltrer les documents non pertinents
✓ Réponse B. Sans overlap, une phrase peut être coupée en deux chunks distincts, perdant son sens. Avec un overlap de 200 caractères (20% de 1000), chaque chunk partage du texte avec son voisin, préservant la continuité sémantique aux jonctions. Typiquement 10-20% du chunk_size.
Question 02 / QCM
Quel est l'avantage principal de l'EnsembleRetriever (BM25 + vectoriel) comparé à chaque approche seule ?
AIl est toujours plus rapide
BCapture à la fois la similarité sémantique et les correspondances de mots-clés exacts
CIl ne nécessite pas de base vectorielle
DIl génère automatiquement des embeddings meilleurs
✓ Réponse B. La recherche vectorielle excelle pour la sémantique (synonymes, paraphrases) mais rate les termes rares et précis. BM25 fait l'inverse. Combinés via Reciprocal Rank Fusion, ils couvrent les cas où chaque méthode échoue seule — le principe fondamental de la recherche hybride.
Question 03 / QCM
Dans le MCP (Model Context Protocol), quelle est la différence entre une Resource et un Tool ?
ALes Resources sont plus rapides que les Tools
CLes Tools ne peuvent pas retourner de données
BResources = données en lecture seule, Tools = actions avec effets de bord
DIl n'y a aucune différence sémantique
✓ Réponse B. Une Resource est l'équivalent d'un endpoint GET : elle expose des données de façon idempotente (fichiers, BDD, configs). Un Tool est l'équivalent d'un POST : il effectue des actions avec des effets de bord potentiels (sauvegarder, envoyer, modifier). Cette distinction sémantique guide l'agent dans son utilisation.

Décrivez en détail comment vous structureriez un système RAG conversationnel capable de référencer des échanges passés dans ses recherches.

2. Mémoire séparée du contexte RAG : L'historique de conversation (ce que l'utilisateur a dit) ≠ le contexte documentaire (ce que le RAG a récupéré). Gérer ces deux flux séparément avec MessagesPlaceholder.

3. Chain finale : create_retrieval_chain(history_aware_retriever, qa_chain) orchestre le tout. La question reformulée alimente le retriever, les docs récupérés alimentent le LLM avec l'historique complet.
