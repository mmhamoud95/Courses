Architecture LangChain

Comprendre ce qu'est LangChain, pourquoi il existe, comment il s'organise et quels composants sont essentiels avant d'écrire la première ligne de code.

Débutant
🎯 Objectif du module

À la fin de ce module, vous comprendrez pourquoi LangChain existe, quelle est la différence entre ses packages, et vous saurez instancier un premier LLM et l'invoquer — en comprenant exactement ce qui se passe sous le capot.

1.1 — Pourquoi LangChain ?

Les LLMs comme GPT-4 ou Llama peuvent générer du texte, mais pour construire une vraie application IA, vous avez besoin de bien plus : connecter le LLM à des données externes, lui donner de la mémoire, orchestrer plusieurs appels, gérer les erreurs... LangChain est le framework qui abstrait toute cette complexité.

🚗

Analogie — Le moteur vs la voiture

Un LLM (GPT, Llama, Mistral) est comme un moteur — très puissant, mais inutilisable seul. LangChain est le châssis de la voiture qui intègre le moteur, la direction, les freins, le tableau de bord. Vous conduisez la voiture, pas le moteur.

✅ Ce que LangChain résout concrètement
  • Connexion aux LLMs : une interface unifiée pour OpenAI, Anthropic, Ollama, Mistral, etc. — changez de LLM en 1 ligne
  • Composition : chaîner prompt → LLM → parser en une expression lisible avec LCEL
  • Mémoire : garder l'historique de conversation entre les appels (les LLMs sont "sans mémoire" nativement)
  • RAG : connecter le LLM à vos données (documents, bases de données, APIs)
  • Agents : permettre au LLM de décider quels outils utiliser pour accomplir une tâche

1.2 — Les 3 packages essentiels

LangChain est divisé en 3 packages distincts — chacun a un rôle précis. Cette séparation évite les dépendances inutiles.

ORGANISATION DES PACKAGES LANGCHAIN langchain-core ← Interfaces abstraites (BaseLanguageModel, BaseRetriever...) └── Pas de LLM réel, pas de DB — juste les contrats └── Stable : ce package change très rarement langchain ← Logique métier (chaînes, agents, mémoire) └── Utilise langchain-core └── Ex: RunnableWithMessageHistory, create_react_agent langchain-community ← Intégrations tierces (Chroma, Redis, DuckDuckGo…) └── Chaque intégration est dans ce package └── Plus volatil — certains outils migrent vers leurs propres packages langchain-ollama ← Package dédié Ollama (modèles locaux) langchain-openai ← Package dédié OpenAI langgraph ← Orchestration d’agents stateful langsmith ← Observabilité, tracing, évaluation

1.3 — Instancier un LLM : LLM vs ChatModel

Il existe deux types de modèles dans LangChain. La distinction est importante car elle détermine le format d'entrée/sortie.

LLM (déprécié)
  • Prend une string en entrée
  • Retourne une string
  • Format : llm.invoke("Bonjour")
  • Modèles : ancienne génération
  • À éviter dans les nouveaux projets
ChatModel (standard actuel)
  • Prend des messages en entrée
  • Retourne un AIMessage
  • Format : llm.invoke([HumanMessage("Bonjour")])
  • Modèles : GPT-4, Claude, Llama3, Mistral
  • Supporte system prompt, historique, tool calling

#  et comment l'invoquer correctement

1.4 — L'interface Runnable — Le contrat universel

Tout dans LangChain implémente l'interface Runnable. C'est le concept le plus important du framework — comprendre Runnable, c'est comprendre 80% de LangChain.

💡 Pourquoi Runnable ?

Un Runnable est tout objet qui peut être "exécuté" avec une entrée pour produire une sortie. Cette interface commune permet de composer n'importe quels éléments ensemble avec le pipe |. Un prompt, un LLM, un parser, un retriever — tous sont des Runnables et peuvent être chaînés.

L'INTERFACE RUNNABLE — 5 MÉTHODES ESSENTIELLES runnable.invoke(input) → Exécute et retourne le résultat (bloquant) runnable.stream(input) → Retourne un générateur de tokens (temps réel) runnable.batch([i1, i2, i3]) → Exécute plusieurs inputs en parallèle runnable.ainvoke(input) → Version async de invoke (non-bloquant) runnable.astream(input) → Version async de stream ───────────────────────────────────────────────────── Tous ces éléments sont des Runnables : ChatOllama → prend des messages, retourne AIMessage ChatPromptTemplate → prend des variables, retourne des messages StrOutputParser → prend un AIMessage, retourne une string VectorStoreRetriever → prend une query, retourne des Documents RunnableLambda(fn) → wrppe n’importe quelle fonction Python
01_runnable.py
python
# ════════════════════════════════════════════════════
#  OBJECTIF : Comprendre l'interface Runnable
#  et les différentes façons d'invoquer un modèle
# ════════════════════════════════════════════════════

from langchain_ollama import ChatOllama
from langchain_core.runnables import RunnableLambda

llm = ChatOllama(model=“llama3.2”)

# ── invoke : résultat complet en une fois ─────────────────────

# Utiliser quand : réponse courte ou traitement après génération

resultat = llm.invoke(“Quelle est la capitale de France ?”)
print(resultat.content)  # “Paris est la capitale de la France.”

# ── stream : tokens en temps réel ────────────────────────────

# Utiliser quand : affichage progressif dans une interface

# Chaque token arrive dès qu’il est généré par le LLM

for chunk in llm.stream(“Raconte une histoire courte :”):
print(chunk.content, end=””, flush=True)  # affiche au fur et à mesure
print()  # saut de ligne à la fin

# ── batch : plusieurs inputs en parallèle ─────────────────────

# Utiliser quand : traiter beaucoup de données rapidement

# LangChain optimise l’exécution en parallèle

questions = [
“Capitale de France ?”,
“Capitale d’Italie ?”,
“Capitale d’Espagne ?”,
]
reponses = llm.batch(questions)  # exécution parallèle automatique
for q, r in zip(questions, reponses):
print(f“Q: {q} → R: {r.content}”)

# ── RunnableLambda : wrapper une fonction Python ──────────────

# Utiliser quand : insérer du code Python custom dans une chaîne

formater = RunnableLambda(lambda texte: texte.upper() + ” !”)
resultat2 = formater.invoke(“bonjour”)
print(resultat2)  # “BONJOUR !”

# async invoke — pour les contextes async (FastAPI, etc.)
import asyncio

async def appel_async():
result = await llm.ainvoke(“Bonjour en async !”)
return result.content

Expliquez avec vos mots ce qu'est l'interface Runnable et pourquoi elle est fondamentale dans LangChain.
✓ Un Runnable est une interface que tous les composants LangChain implémentent. Elle garantit que tout composant peut être invoqué avec .invoke(), streamé avec .stream(), exécuté en batch avec .batch(). Ce contrat commun est ce qui rend la composition possible : puisque tout est Runnable, on peut chaîner n'importe quoi avec le pipe | (LCEL). Sans cette abstraction, chaque composant aurait une API différente et la composition serait impossible.

LCEL — LangChain Expression Language

La façon moderne de composer des chaînes dans LangChain. LCEL permet d'assembler des composants avec le symbole | (pipe), comme dans un shell Unix.

Débutant
🎯 Objectif du module

Comprendre ce qu'est LCEL, pourquoi c'est la façon recommandée de construire des chaînes, et maîtriser toutes les façons de composer et d'invoquer des chaînes complexes — séquentielles, parallèles et avec branches.

2.1 — Qu'est-ce que LCEL et pourquoi l'utiliser ?

Avant LCEL (2023), construire une chaîne dans LangChain nécessitait des classes comme LLMChain, SequentialChain, etc. — verbeuses et difficiles à déboguer. LCEL remplace tout ça par une syntaxe déclarative simple : le pipe |.

🐧

Analogie — Le pipe Unix

En shell, cat fichier.txt | grep "erreur" | sort | uniq chaîne des commandes. Chaque commande prend la sortie de la précédente. LCEL fonctionne exactement pareil : prompt | llm | parser.

✅ Avantages de LCEL
  • Lisibilité : le flux de données se lit de gauche à droite, comme on pense
  • Streaming automatique : si vous appelez .stream() sur une chaîne, chaque composant streame automatiquement
  • Parallélisme gratuit : RunnableParallel exécute plusieurs branches en parallèle sans code supplémentaire
  • Tracing intégré : LangSmith trace automatiquement chaque étape d'une chaîne LCEL
  • Composable : une chaîne LCEL est elle-même un Runnable — elle peut être insérée dans une autre chaîne

2.2 — Le pipe | — Construction d'une chaîne simple

Ce que fait le pipe |

L'expression A | B crée un nouveau Runnable qui : prend l'input, le passe à A, prend la sortie de A, la passe à B, et retourne la sortie de B. C'est équivalent à B(A(input)) mais beaucoup plus lisible.

La chaîne la plus commune : prompt | llm | parser

C'est la chaîne de base de 90% des applications LangChain. Le prompt prépare les messages, le LLM génère une réponse, le parser extrait le texte brut.

02_lcel_base.py
python
# ════════════════════════════════════════════════════
#  OBJECTIF : Construire votre première chaîne LCEL
#  et comprendre ce que fait chaque étape
# ════════════════════════════════════════════════════

from langchain_ollama import ChatOllama
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

llm = ChatOllama(model=“llama3.2”, temperature=0)

# ── Étape 1 : Créer un prompt template ─────────────────────────

# ChatPromptTemplate.from_template prend une string avec des {variables}

# Ces variables seront remplacées lors de l’invocation

prompt = ChatPromptTemplate.from_template(
“Explique {concept} en une seule phrase simple.”
)

# ── Étape 2 : Créer un parser ───────────────────────────────────

# StrOutputParser transforme un AIMessage en string Python pure

# Sans lui, llm.invoke() retourne un objet AIMessage (pas une string)

parser = StrOutputParser()

# ── Étape 3 : Assembler la chaîne avec | ───────────────────────

# prompt reçoit {“concept”: “…”} → retourne des messages

# llm    reçoit des messages        → retourne AIMessage

# parser reçoit AIMessage           → retourne string

chain = prompt | llm | parser

# ── Étape 4 : Invoquer la chaîne ───────────────────────────────

# On passe un dict avec les variables du template

resultat = chain.invoke({“concept”: “l’apprentissage automatique”})
print(type(resultat))   #  — grâce au StrOutputParser
print(resultat)         # “L’apprentissage automatique est…”

# ── Pour comprendre : sans le parser ───────────────────────────
chain_sans_parser = prompt | llm
resultat2 = chain_sans_parser.invoke({“concept”: “Python”})
print(type(resultat2))  # AIMessage — pas une string !
print(resultat2.content)# Il faut accéder à .content
Ce qui se passe à chaque étape
Entrée → {"concept": "l'apprentissage automatique"}
prompt → [SystemMessage, HumanMessage("Explique l'apprentissage automatique...")]
llm → AIMessage(content="L'apprentissage automatique est...")
parser → "L'apprentissage automatique est..." # string Python pure

Pourquoi stream() est-il préférable à invoke() pour une interface utilisateur ?
✓ invoke() attend que toute la réponse soit générée (peut durer 5-30 secondes) avant de retourner quoi que ce soit à l'interface — l'utilisateur voit un écran vide pendant ce temps, ce qui est frustrant. stream() retourne chaque token dès qu'il est disponible, permettant à l'interface d'afficher le texte progressivement — comme ChatGPT. 4.1 — Pourquoi les LLMs n'ont pas de mémoire ?

📞

Analogie — L'amnésique au téléphone

Imaginez quelqu'un qui perd la mémoire entre chaque phrase d'une conversation. Vous dites "Mon nom est Alice", il répond "Bonjour Alice". Puis vous demandez "Quel est mon nom ?" — et il a tout oublié. Les LLMs sont comme ça : chaque appel est indépendant, le modèle ne "souvient" de rien du tout entre deux invocations.

RAG — Retrieval-Augmented Generation

Connecter le LLM à vos propres données : documents PDF, bases de données, sites web — pour répondre à des questions sur des informations inconnues du modèle.

Intermédiaire

5.1 — Qu'est-ce que le RAG et pourquoi en a-t-on besoin ?

💡 Le problème que résout le RAG

Les LLMs ont une date de coupure (ils ne connaissent pas les infos récentes) et n'ont pas accès à vos données internes (docs d'entreprise, wiki, support). Le RAG résout ces deux problèmes en récupérant dynamiquement les informations pertinentes avant de générer la réponse.

📖

Analogie — L'examen à livre ouvert

Le LLM sans RAG = examen de mémoire (le modèle ne sait que ce qu'il a appris). Le LLM avec RAG = examen à livre ouvert — avant de répondre, il consulte les documents pertinents. La différence : qualité et précision des réponses.

PIPELINE RAG — 6 ÉTAPES PHASE INDEXATION (une seule fois) 1. Chargement → PDFs, URLs, textes bruts [DocumentLoaders] 2. Découpage → Chunks de 1000 chars + 200 overlap [TextSplitters] 3. Embedding → Chaque chunk → vecteur numérique [EmbeddingModel] 4. Stockage → Vecteurs dans ChromaDB/FAISS [VectorStore] PHASE RETRIEVAL + GÉNÉRATION (à chaque question) 5. Retrieval → Question → top-K chunks similaires [Retriever] 6. Génération → Chunks + Question → Réponse finale [LLM + Prompt]

5.2 — Étapes 1-4 : Indexation des documents

Étape 1 — DocumentLoaders : charger vos données

LangChain fournit des loaders pour presque tout format : PDF, Markdown, CSV, HTML, DOCX, YouTube, URLs, GitHub... Chaque loader retourne une liste d'objets Document avec page_content (le texte) et metadata (source, page, etc.).

Étape 2 — TextSplitters : découper les documents

Les LLMs ont une fenêtre de contexte limitée — on ne peut pas leur envoyer tout le document. On découpe en chunks avec un chevauchement (overlap) pour ne pas perdre le contexte aux jonctions. Le RecursiveCharacterTextSplitter est le meilleur choix pour la plupart des cas.

Étape 3 — Embeddings : transformer le texte en vecteurs

Un modèle d'embedding transforme du texte en un tableau de nombres (vecteur). Des textes similaires ont des vecteurs proches. C'est la technologie qui permet la recherche sémantique.

Étape 4 — VectorStore : stocker et rechercher les vecteurs

Le vectorstore stocke les vecteurs et permet une recherche par similarité ultra-rapide. Chroma est idéal en développement (local, persisté sur disque). Pour la production : Pinecone, Qdrant, Weaviate.

7.1 — LangChain vs LangGraph

LangChain LCEL seul
  • Flux linéaire : A → B → C
  • Pas de boucles nativement
  • Pas d'état partagé entre étapes
  • Difficile de brancher conditionnellement
  • Parfait pour les chaînes simples
LangGraph
  • Graphe : nœuds + arêtes conditionnelles
  • Boucles natives (réessayer, affiner)
  • État TypedDict partagé entre tous les nœuds
  • Branches selon la sortie d'un nœud
  • Human-in-the-Loop avec pause/reprise
🗺

Analogie — La recette vs le GPS

LCEL est une recette : des étapes linéaires fixes. LangGraph est un GPS : il a une carte (le graphe), sait où il est (l'état), prend des décisions selon les conditions (embouteillage → recalculer), peut revenir en arrière. Le GPS s'adapte, la recette suit toujours le même chemin.

7.2 — Les 3 concepts fondamentaux : State, Nodes, Edges

State — L'état partagé (TypedDict)

Le State est un TypedDict qui contient toutes les données partagées entre nœuds. Chaque nœud peut le lire et le modifier. Annotated[list, operator.add] signifie "accumuler" (append) au lieu de remplacer — essentiel pour l'historique de messages.

Nodes — Les nœuds (fonctions Python)

Un nœud est une fonction Python ordinaire qui prend l'état en entrée et retourne un dict de modifications. LangGraph applique ces modifications à l'état global. Chaque nœud ne connaît que sa partie du travail.

Edges — Les arêtes (connexions)

add_edge("A", "B") = A appelle toujours B. add_conditional_edges("A", fn) = A routing dynamique selon la fonction fn. La fonction retourne une string qui correspond à un nœud ou à END.

STRUCTURE D'UN LANGGRAPH StateGraph(MonState) │ ├── add_node("noeud_a", fonction_a) ├── add_node("noeud_b", fonction_b) ├── add_node("noeud_c", fonction_c) │ ├── set_entry_point("noeud_a") ← point d'entrée │ ├── add_edge("noeud_a", "noeud_b") ← arête fixe : a→b toujours │ └── add_conditional_edges( ← arête conditionnelle "noeud_b", ma_fonction_routing, ← retourne "noeud_c" ou "fin" {"noeud_c": "noeud_c", "fin": END} ) │ └── compile() ← compile en Runnable (peut être invoqué)

from langgraph.graph import StateGraph, END
from langgraph.checkpoint.memory import MemorySaver
from langchain_core.messages import BaseMessage, HumanMessage, AIMessage
from langchain_ollama import ChatOllama
from typing import TypedDict, Annotated, Literal
import operator

llm = ChatOllama(model=“llama3.2”, temperature=0)

# ── 1. Définir l’état ─────────────────────────────────────────────

# TypedDict : contrat strict entre les nœuds

# Annotated[list, operator.add] : chaque nœud AJOUTE à la liste

# (au lieu de remplacer la liste entière)

class AgentState(TypedDict):
messages:  Annotated[list[BaseMessage], operator.add]
iteration: int   # compteur pour éviter boucles infinies
done:      bool  # signal d’arrêt

# ── 2. Définir les nœuds ──────────────────────────────────────────

# Chaque nœud = fonction(state) → dict de modifications

def call_llm(state: AgentState) -> dict:
””“Appelle le LLM avec tous les messages de l’état.”””
print(f”  [call_llm] Tour {state[‘iteration’] + 1}”)
response = llm.invoke(state[“messages”])
return {
“messages”:  [response],               # ajouté à la liste existante
“iteration”: state[“iteration”] + 1,  # incrémenté
}

def check_quality(state: AgentState) -> dict:
””“Vérifie si la dernière réponse est satisfaisante.”””
last_content = state[“messages”][-1].content
# Critère simple : réponse suffisamment longue
is_done = len(last_content) >= 100
print(f”  [check_quality] {len(last_content)} chars → {‘OK’ if is_done else ‘trop court’}”)
return {“done”: is_done}

# ── 3. Fonction de routing ────────────────────────────────────────

# Retourne une string → correspond à un nœud ou END

def should_continue(state: AgentState) -> Literal[“retry”, “end”]:
””“Décide si on continue ou s’arrête.”””
if state[“done”] or state[“iteration”] >= 4:
return “end”
return “retry”

# ── 4. Construire le graphe ───────────────────────────────────────
graph = StateGraph(AgentState)

graph.add_node(“llm”,     call_llm)
graph.add_node(“checker”, check_quality)

graph.set_entry_point(“llm”)           # entrée du graphe
graph.add_edge(“llm”, “checker”)        # fixe : llm → checker
graph.add_conditional_edges(             # conditionnel : checker → ?
“checker”,
should_continue,
{
“retry”: “llm”,  # → relancer le LLM
“end”:   END,    # → terminer
}
)

app = graph.compile()

# Visualiser le graphe
app.get_graph().print_ascii()

result = app.invoke({
“messages”:  [HumanMessage(“Explique LangGraph en détail avec exemples.”)],
“iteration”: 0,
“done”:      False,
})
print(f”\n✅ Réponse finale ({result[‘iteration’]} tour(s)):”)
print(result[“messages”][-1].content)

8.1 — Pourquoi plusieurs agents spécialisés ?

💡 La règle d'or : moins de tools par agent = meilleures décisions

Un agent avec 20 tools différents se trompe souvent de tool. Un agent avec 3-5 tools ciblés est beaucoup plus précis. Le superviseur fait office de chef d'équipe : il délègue au bon spécialiste selon le contexte, comme un manager qui sait quelle expertise mobiliser.

# ── 2. État partagé ────────────────────────────────────────────────
class SuperState(TypedDict):
messages:  Annotated[list, operator.add]
next_node: str  # quel agent le superviseur choisit

# ── 3. Schéma de décision structuré ───────────────────────────────

# with_structured_output → le LLM répond en JSON validé par Pydantic

class Decision(BaseModel):
next: Literal[“math”, “meteo”, “recherche”, “FINISH”] = Field(
description=””“Choisir l’agent approprié :
- math       : opérations numériques, calculs, statistiques
- meteo      : météo, température, date/heure actuelle
- recherche  : actualités, infos récentes, faits généraux
- FINISH     : la question est entièrement résolue, répondre à l’utilisateur”””
)
raison: str = Field(description=“Courte justification du choix”)

supervisor_llm = llm.with_structured_output(Decision)

# ── 4. Nœud superviseur ────────────────────────────────────────────
def supervisor(state: SuperState) -> dict:
””“Analyse l’historique et décide quel agent appeler.”””
# Résumé des 6 derniers messages pour ne pas saturer le contexte
hist = ”\n”.join([
f”[{m.**class**.**name**}]: {m.content[:120]}”
for m in state[“messages”][-6:]
])
decision = supervisor_llm.invoke(
f””“Tu coordonnes 3 agents : math, meteo, recherche.
Historique de la conversation :
{hist}

Qui doit agir maintenant ? Réponds FINISH si tout est résolu.”””
)
print(f“🎯 Supervisor → {decision.next} | {decision.raison}”)
return {“next_node”: decision.next}

# ── 5. Wrapper : exécuter un sous-agent ────────────────────────────
def make_agent_node(agent, name: str):
””“Crée un nœud LangGraph qui exécute un sous-agent.”””
def node(state: SuperState) -> dict:
print(f”  🤖 Agent [{name}] en cours…”)
result = agent.invoke({“messages”: state[“messages”]})
last   = result[“messages”][-1]
# Retourner la réponse avec le nom de l’agent comme attribut
return {“messages”: [AIMessage(content=last.content, name=name)]}
node.**name** = name
return node

# ── 6. Construire le graphe ────────────────────────────────────────
g = StateGraph(SuperState)

g.add_node(“supervisor”, supervisor)
g.add_node(“math”,       make_agent_node(agent_math,   “math”))
g.add_node(“meteo”,      make_agent_node(agent_meteo,  “meteo”))
g.add_node(“recherche”,  make_agent_node(agent_search, “recherche”))

g.set_entry_point(“supervisor”)

# Superviseur → agent choisi (ou END)
g.add_conditional_edges(“supervisor”, lambda s: s[“next_node”], {
“math”:      “math”,
“meteo”:     “meteo”,
“recherche”: “recherche”,
“FINISH”:    END,
})
# Chaque agent retourne toujours au superviseur
for node_name in [“math”, “meteo”, “recherche”]:
g.add_edge(node_name, “supervisor”)

multi_agent_app = g.compile()

# ── 7. Test ───────────────────────────────────────────────────────
result = multi_agent_app.invoke({
“messages”:  [HumanMessage(“Combien fait 17 * 23 ? Et quelle heure est-il ?”)],
“next_node”: “supervisor”,
})
print(”\n✅ Réponse finale:”, result[“messages”][-1].content)
