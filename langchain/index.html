1.2 — L'interface Runnable (cœur de LCEL)

💡

Tout est Runnable

Dans LangChain moderne, chaque composant (LLM, prompt, parser, retriever…) implémente l'interface Runnable. Cela garantit une API unifiée : invoke, stream, batch, ainvoke. L'opérateur | compose des Runnables en pipeline.

from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import (
RunnablePassthrough,
RunnableLambda,
RunnableParallel,
RunnableBranch,
)

llm    = ChatOllama(model=“llama3.2”)
prompt = ChatPromptTemplate.from_messages([
(“system”, “Tu es un assistant concis.”),
(“human”,  ”{question}”),
])

# ── 1. Chain simple ─────────────────────────────────────────
chain = prompt | llm | StrOutputParser()

chain.invoke({“question”: “Qu’est-ce que Python?”})   # str
chain.batch([{“question”: “Python?”}, {“question”: “Rust?”}])  # parallèle
for c in chain.stream({“question”: “Explique les LLMs”}):
print(c, end=””, flush=True)                       # streaming

# ── 2. RunnablePassthrough — passer sans modifier ────────────
chain_rag = {
“contexte”: RunnablePassthrough(),   # garde la valeur intacte
“question”: RunnablePassthrough(),
}

# ── 3. RunnablePassthrough.assign — ajouter des clés ────────
enriched = (
RunnablePassthrough.assign(
topic_upper=lambda x: x[“topic”].upper()
)
| prompt | llm | StrOutputParser()
)

# ── 4. RunnableLambda — fonction Python classique ────────────
def add_context(x: dict) -> dict:
x[“context”] = f”Contexte: {x[‘topic’]}”
return x

chain_with_fn = RunnableLambda(add_context) | prompt | llm | StrOutputParser()

# ── 5. RunnableParallel — deux branches simultanées ──────────
parallel = RunnableParallel({
“resume”:   chain,
“question”: RunnablePassthrough(),
})
r = parallel.invoke({“question”: “Explique les LLMs”})
# r = {“resume”: “…”, “question”: {“question”: “…”}}

# ── 6. RunnableBranch — routing conditionnel ─────────────────
branch = RunnableBranch(
(lambda x: “code”  in x[“question”].lower(), chain),
(lambda x: “math”  in x[“question”].lower(), chain),
chain,   # default
)

# ── 7. Robustesse ────────────────────────────────────────────
safe_chain = chain.with_retry(stop_after_attempt=3)
chain_fb   = chain.with_fallbacks([ChatOllama(model=“mistral”) | StrOutputParser()])
cached     = chain.with_config({“run_name”: “mon_pipeline”})  # LangSmith

02_async.py

import asyncio
```

# ainvoke — asynchrone
async def run():
result = await chain.ainvoke({“question”: “Bonjour”})
return result

# astream — streaming async (idéal pour FastAPI / WebSockets)
async def stream_response():
async for chunk in chain.astream({“question”: “Explique les transformers”}):
yield chunk   # parfait pour SSE dans FastAPI

# abatch — batch asynchrone avec limite de concurrence
async def batch_async():
questions = [{“question”: f“Qu’est-ce que {t}?”} for t in [“Python”, “Rust”, “Go”]]
results = await chain.abatch(questions, config={“max_concurrency”: 3})
return results

asyncio.run(run())

Q1 · QCM
Que retourne chain.batch([input1, input2, input3]) par rapport à 3 appels invoke séparés ?
AExactement la même chose
BExécute les 3 en parallèle — bien plus rapide
CLes fusionne en un seul appel LLM
DN'est utile que pour OpenAI

Q2 · QCM
Quel Runnable permet d'exécuter deux branches simultanément et de fusionner leurs résultats dans un dict ?
ARunnableLambda
BRunnableBranch
CRunnableParallel
DRunnablePassthrough
✓ C. RunnableParallel({"a": chainA, "b": chainB}) exécute les deux branches en parallèle et retourne {"a": résultatA, "b": résultatB}.
Q3 · Ouverte
Quelle est la différence entre RunnableBranch et RunnableParallel ?
✓ RunnableParallel exécute toutes les branches simultanément et retourne tous les résultats. RunnableBranch évalue les conditions dans l'ordre et n'exécute que la première branche dont la condition est vraie — c'est du routing, pas du parallélisme.

from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.output_parsers import StrOutputParser
from langchain_core.chat_history import InMemoryChatMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory

llm = ChatOllama(model=“llama3.2”)

prompt = ChatPromptTemplate.from_messages([
(“system”, “Tu es un assistant serviable. Utilise l’historique.”),
MessagesPlaceholder(“history”),   # historique injecté ici
(“human”,  ”{input}”),
])
chain = prompt | llm | StrOutputParser()

# ── Stockage en mémoire par session ─────────────────────────
store: dict[str, InMemoryChatMessageHistory] = {}

def get_history(session_id: str) -> InMemoryChatMessageHistory:
if session_id not in store:
store[session_id] = InMemoryChatMessageHistory()
return store[session_id]

chain_mem = RunnableWithMessageHistory(
chain,
get_history,
input_messages_key=“input”,
history_messages_key=“history”,
)

# ── Utilisation ──────────────────────────────────────────────
cfg_alice = {“configurable”: {“session_id”: “alice”}}

r1 = chain_mem.invoke({“input”: “Je m’appelle Alice.”}, cfg_alice)

2 questions
Q1 · QCM
Pourquoi l'EnsembleRetriever (BM25 + vectoriel) est-il souvent meilleur que chaque méthode seule ?
AIl est plus rapide que chaque méthode
BBM25 capture les mots-clés exacts, le vectoriel capture la sémantique
CIl génère automatiquement de meilleurs embeddings
DIl ne nécessite pas de base vectorielle
✓ B. BM25 (TF-IDF) excelle pour les termes rares et précis. La recherche vectorielle capture les synonymes et la sémantique. Combinés via Reciprocal Rank Fusion, ils couvrent les cas où chaque méthode échoue seule.
Q2 · Ouverte
Expliquez le rôle de chunk_overlap et donnez la valeur empirique recommandée.
✓ Sans overlap, une phrase coupée entre deux chunks perd son sens à la jonction. Avec overlap, chaque chunk partage du texte avec son voisin — la continuité sémantique est préservée. Règle empirique : 10-20% du chunk_size. Pour chunk_size=1000, utilisez chunk_overlap=200.

from pydantic import BaseModel, Field
from datetime import datetime
import math, os

# ── @tool — décorateur simple (recommandé) ───────────────────
@tool
def calculatrice(expression: str) -> str:
””“Évalue une expression mathématique Python sécurisée.
Exemples: “2 + 2”, “100 * 0.15”, “sqrt(144)”
“””
try:
env = {k: getattr(math, k) for k in dir(math) if not k.startswith(”_”)}
env[”**builtins**”] = {}
return str(eval(expression, env))
except Exception as e:
return f“Erreur: {e}”

@tool
def date_heure() -> str:
””“Retourne la date et heure actuelles.”””
return datetime.now().strftime(”%A %d %B %Y à %H:%M”)

@tool
def lire_fichier(chemin: str) -> str:
””“Lit le contenu d’un fichier texte local.
Args: chemin — chemin relatif du fichier
“””
try:
with open(chemin, encoding=“utf-8”) as f: return f.read()[:2000]
except: return f“Fichier non trouvé: {chemin}”

@tool
def ecrire_fichier(chemin: str, contenu: str) -> str:
””“Écrit du contenu dans un fichier. Crée les répertoires si nécessaire.”””
os.makedirs(os.path.dirname(chemin) or ”.”, exist_ok=True)
with open(chemin, “w”, encoding=“utf-8”) as f: f.write(contenu)
return f“✅ Écrit: {chemin} ({len(contenu)} chars)”

# ── StructuredTool — schéma Pydantic explicite ───────────────
class ConvertInput(BaseModel):
valeur:     float = Field(description=“Valeur à convertir”)
de_unite:   str   = Field(description=“Unité source (km/miles/kg/lbs)”)
vers_unite: str   = Field(description=“Unité cible”)

def _convert(valeur, de_unite, vers_unite):
table = {(“km”,“miles”):.621,(“miles”,“km”):1.609,(“kg”,“lbs”):2.205,(“lbs”,“kg”):.454}
f = table.get((de_unite.lower(), vers_unite.lower()))
return f”{valeur*f:.3f} {vers_unite}” if f else “Conversion non supportée”

conversion = StructuredTool.from_function(
func=_convert, name=“conversion”,
description=“Convertit entre unités (km/miles, kg/lbs)”,
args_schema=ConvertInput
)

# Inspecter un tool
print(calculatrice.name, calculatrice.description)
print(calculatrice.invoke({“expression”: “sqrt(144)”}))  # “12.0”

Pourquoi LangGraph plutôt qu'AgentExecutor ?

AgentExecutor est une boucle opaque. LangGraph vous donne le contrôle total : définissez explicitement les nœuds (fonctions Python), les arêtes (transitions conditionnelles), l'état partagé, et les cycles. Indispensable pour les workflows complexes et la production.

7.1 — Concepts fondamentaux

ConceptDescriptionAnalogie
StateGraphGraphe avec état partagé entre les nœudsMachine à états finis
State (TypedDict)Dict typé partagé entre tous les nœudsBase de données en mémoire
NodeFonction Python qui reçoit et retourne l'étatÉtape de traitement
EdgeTransition fixe entre deux nœudsFlèche dans un diagramme
Conditional EdgeTransition conditionnelle basée sur l'étatif/elif dans le flux
CheckpointerPersistance de l'état entre les runsSave game
InterruptPause pour intervention humaineHuman-in-the-loop

from langgraph.graph import StateGraph, END
from langgraph.prebuilt import ToolNode, tools_condition
from langchain_ollama import ChatOllama
from langchain_core.messages import BaseMessage, HumanMessage
from typing import TypedDict, Annotated
import operator

# ── 1. Définir l’état partagé ────────────────────────────────
class AgentState(TypedDict):
# operator.add → les messages s’accumulent (pas de remplacement)
messages: Annotated[list[BaseMessage], operator.add]

# ── 2. LLM avec tools bindés ─────────────────────────────────
llm   = ChatOllama(model=“llama3.2”)
tools = [calculatrice, date_heure, lire_fichier]
llm_t = llm.bind_tools(tools)

# ── 3. Nœud agent ────────────────────────────────────────────
def call_agent(state: AgentState) -> dict:
return {“messages”: [llm_t.invoke(state[“messages”])]}

# ── 4. Nœud tools (automatique) ──────────────────────────────
tool_node = ToolNode(tools)

# ── 5. Construire le graphe ───────────────────────────────────
graph_builder = StateGraph(AgentState)
graph_builder.add_node(“agent”, call_agent)
graph_builder.add_node(“tools”, tool_node)

graph_builder.set_entry_point(“agent”)

# Condition: si le LLM veut un tool → “tools”, sinon → END
graph_builder.add_conditional_edges(
“agent”, tools_condition,
{“tools”: “tools”, END: END}
)
graph_builder.add_edge(“tools”, “agent”)  # après tools → retour agent

graph = graph_builder.compile()

# ── 6. Exécuter ──────────────────────────────────────────────
result = graph.invoke({
“messages”: [HumanMessage(“Quelle date? Et calcule sqrt(196).”)]
})
print(result[“messages”][-1].content)

# Workflow : question → classifier → spécialiste → réponse

from typing import Literal

class RouterState(TypedDict):
question:  str
categorie: str
reponse:   str

# ── Nœuds ────────────────────────────────────────────────────
def classifier(state: RouterState) -> dict:
””“Classifie la question.”””
prompt = f“Classifie en UN mot (code/math/general): {state[‘question’]}”
result = llm.invoke(prompt).content.strip().lower()
return {“categorie”: result}

def expert_code(state: RouterState) -> dict:
r = llm.invoke(f“Expert Python. {state[‘question’]}”)
return {“reponse”: r.content}

def expert_math(state: RouterState) -> dict:
r = llm.invoke(f“Expert maths. Explique chaque étape. {state[‘question’]}”)
return {“reponse”: r.content}

def expert_general(state: RouterState) -> dict:
r = llm.invoke(state[“question”])
return {“reponse”: r.content}

# ── Condition de routage ──────────────────────────────────────
def route(state: RouterState) -> Literal[“expert_code”, “expert_math”, “expert_general”]:
cat = state[“categorie”]
if “code”   in cat: return “expert_code”
if “math”   in cat: return “expert_math”
return “expert_general”

# ── Construction ─────────────────────────────────────────────
g = StateGraph(RouterState)
for name, fn in [(“classifier”,classifier),(“expert_code”,expert_code),
(“expert_math”,expert_math),(“expert_general”,expert_general)]:
g.add_node(name, fn)

g.set_entry_point(“classifier”)
g.add_conditional_edges(“classifier”, route)
g.add_edge(“expert_code”,    END)
g.add_edge(“expert_math”,    END)
g.add_edge(“expert_general”, END)

router_graph = g.compile()
r = router_graph.invoke({“question”: “Écris une fonction de tri en Python”})
print(r[“reponse”])

from langgraph.graph import StateGraph, END
from langgraph.prebuilt import create_react_agent
from langchain_core.messages import HumanMessage, AIMessage
from typing import TypedDict, Annotated, Literal
import operator

# ── Sous-agents spécialisés ──────────────────────────────────
agent_recherche = create_react_agent(llm, [lire_fichier],   # accès docs
state_modifier=“Tu es un spécialiste en recherche documentaire.”)
agent_calcul    = create_react_agent(llm, [calculatrice],   # calculs
state_modifier=“Tu es un expert en analyse chiffrée.”)
agent_redaction = create_react_agent(llm, [ecrire_fichier], # rédaction
state_modifier=“Tu es un expert en rédaction de rapports.”)

# ── État partagé ─────────────────────────────────────────────
class SuperState(TypedDict):
messages: Annotated[list, operator.add]
next:     str   # prochain agent à appeler

# ── Superviseur ──────────────────────────────────────────────
AGENTS = [“recherche”, “calcul”, “redaction”, “FINISH”]

def supervisor(state: SuperState) -> dict:
””“Décide quel agent appeler ou si on a terminé.”””
history = ”\n”.join(m.content for m in state[“messages”][-5:])
decision = llm.invoke(f”””
Agents: {AGENTS}
Historique: {history}
Réponds avec UN seul agent à appeler (ou FINISH si terminé).
Agent:”””).content.strip()

```
next_agent = next((a for a in AGENTS if a.lower() in decision.lower()), "FINISH")
return {"next": next_agent}
```

def make_agent_node(agent, name: str):
def node(state: SuperState) -> dict:
result = agent.invoke({“messages”: state[“messages”]})
return {“messages”: [AIMessage(content=result[“messages”][-1].content,
name=name)]}
return node

# ── Construire le graphe superviseur ─────────────────────────
g = StateGraph(SuperState)
g.add_node(“supervisor”,  supervisor)
g.add_node(“recherche”,   make_agent_node(agent_recherche, “recherche”))
g.add_node(“calcul”,      make_agent_node(agent_calcul,    “calcul”))
g.add_node(“redaction”,   make_agent_node(agent_redaction,  “redaction”))

g.set_entry_point(“supervisor”)

# Le superviseur route vers le bon agent
g.add_conditional_edges(“supervisor”, lambda x: x[“next”], {
“recherche”: “recherche”,
“calcul”:    “calcul”,
“redaction”: “redaction”,
“FINISH”:    END,
})
# Chaque agent renvoie au superviseur
for node in [“recherche”, “calcul”, “redaction”]:
g.add_edge(node, “supervisor”)

super_graph = g.compile(checkpointer=MemorySaver())

cfg = {“configurable”: {“thread_id”: “task_1”}}
result = super_graph.invoke({
“messages”: [HumanMessage(“Recherche les ventes Q3, calcule la croissance vs Q2, puis rédige un résumé.”)]
}, cfg)
print(result[“messages”][-1].content)

g.add_conditional_edges("agent", tools_condition, {"tools":"tools", END:END})
g.add_edge("tools", "agent")

with SqliteSaver.from_conn_string("data/checkpoints.db") as saver:
    return g.compile(checkpointer=saver)
