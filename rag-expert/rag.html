Module 1 · Fondation

Reranking — Cross-Encoder

Améliorer l'ordre des chunks récupérés avec un modèle de reranking, pour donner au LLM les passages les plus pertinents en premier.

Intermédiaire

1.1 — Le problème que résout le Reranking

❌ Sans Reranking
  • Le retriever retourne les K chunks les plus similaires
  • Similarité ≠ pertinence réelle pour la question
  • Le chunk le plus utile peut être en position 4 ou 5
  • Le LLM souffre de "lost in the middle" — il ignore les passages au milieu
✅ Avec Reranking
  • Un 2ème modèle classe les chunks par pertinence réelle
  • Les chunks les plus utiles remontent en tête
  • On peut réduire k (ex: 10→3) sans perte de qualité
  • Le LLM reçoit un contexte plus dense et pertinent

1.2 — Bi-Encoder vs Cross-Encoder

CritèreBi-Encoder (retriever)Cross-Encoder (reranker)
FonctionnementEncode query et doc séparément → similarité cosinusEncode query + doc ensemble → score de pertinence
VitesseUltra-rapide (pré-calculé)Lent (calcul à la volée)
PrécisionBonneExcellente
UsageRécupération initiale (top-20)Reclassement final (top-20 → top-3)
Modèlesnomic-embed-text, all-MiniLMBGE-Reranker, ms-marco-MiniLM
🎯

Pattern Pareto — Retrieve large, Rerank small

Récupérez 20-50 candidats avec le retriever rapide, puis rerankez pour ne garder que les 3-5 meilleurs. C'est le meilleur ratio qualité/latence pour 90% des cas.

1.3 — Implémentation avec sentence-transformers

01_reranking.py
python 
# pip install sentence-transformers langchain langchain-ollama chromadb
from sentence_transformers import CrossEncoder
from langchain_ollama import OllamaEmbeddings, ChatOllama
from langchain_community.vectorstores import Chroma
from langchain_core.documents import Document
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

# ── Charger le Cross-Encoder ────────────────────────────────────
# Modèles recommandés (par ordre de qualité/vitesse):
# - BAAI/bge-reranker-base     : rapide, très bon
# - cross-encoder/ms-marco-MiniLM-L-6-v2 : ultra-léger
# - BAAI/bge-reranker-large    : meilleure qualité
reranker = CrossEncoder(“BAAI/bge-reranker-base”)

# ── Setup vectorstore ───────────────────────────────────────────
embeddings  = OllamaEmbeddings(model=“nomic-embed-text”)
vectorstore = Chroma(persist_directory=”./chroma_db”, embedding_function=embeddings)

# ── Fonction de reranking ───────────────────────────────────────
def rerank(query: str, docs: list[Document], top_k: int = 3) -> list[Document]:
””“Rerank les documents par pertinence réelle avec le cross-encoder.”””
if not docs:
return []

```
# Créer les paires (query, passage) pour le cross-encoder
pairs = [(query, doc.page_content) for doc in docs]

# Calculer les scores de pertinence
scores = reranker.predict(pairs)

# Trier par score décroissant
ranked = sorted(zip(scores, docs), key=lambda x: x[0], reverse=True)

# Retourner les top_k documents avec leur score
result = []
for score, doc in ranked[:top_k]:
    doc.metadata["rerank_score"] = round(float(score), 4)
    result.append(doc)

return result
```

# ── Test standalone ─────────────────────────────────────────────
query    = “Quelle est la politique de remboursement?”
# Récupérer 20 candidats avec le retriever rapide
candidates = vectorstore.similarity_search(query, k=20)
# Reranker pour obtenir les 3 meilleurs
best_docs  = rerank(query, candidates, top_k=3)

for doc in best_docs:
print(f“Score: {doc.metadata[‘rerank_score’]:.4f} | {doc.page_content[:80]}…”)

1.4 — Intégration dans une chain LCEL

01_rerank_chain.py
python 
from langchain_core.runnables import RunnableLambda, RunnableParallel

llm = ChatOllama(model=“llama3.2”, temperature=0)

# ── Retriever avec reranking intégré ────────────────────────────
class RerankRetriever:
””“Retriever qui applique automatiquement le reranking.”””

```
def __init__(self, vectorstore, reranker, fetch_k=20, top_k=4):
    self.vs       = vectorstore
    self.reranker = reranker
    self.fetch_k  = fetch_k
    self.top_k    = top_k

def get_relevant_documents(self, query: str) -> list[Document]:
    # 1. Récupérer beaucoup de candidats
    candidates = self.vs.similarity_search(query, k=self.fetch_k)
    # 2. Reranker pour garder les meilleurs
    return rerank(query, candidates, top_k=self.top_k)

def __call__(self, query: str) -> list[Document]:
    return self.get_relevant_documents(query)
```

rerank_retriever = RerankRetriever(vectorstore, reranker, fetch_k=20, top_k=4)

# ── Chain RAG avec reranking ─────────────────────────────────────
def fmt_docs(docs):
return ”\n\n—\n\n”.join(
f”[Score: {d.metadata.get(‘rerank_score’, ‘N/A’)} | {d.metadata.get(‘source’,’?’)}]\n{d.page_content}”
for d in docs
)

rag_prompt = ChatPromptTemplate.from_template(”””
Réponds uniquement en te basant sur le contexte ci-dessous.

Contexte (trié par pertinence):
{context}

Question: {question}
Réponse:”””)

# La chain complète avec reranking transparent
rag_chain = (
{
“context”:  RunnableLambda(rerank_retriever) | fmt_docs,
“question”: RunnablePassthrough(),
}
| rag_prompt | llm | StrOutputParser()
)

answer = rag_chain.invoke(“Quelle est la politique de remboursement?”)
print(answer)

# ── Alternative: ContextualCompressionRetriever (LangChain natif) ─
from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import CrossEncoderReranker
from langchain_community.cross_encoders import HuggingFaceCrossEncoder

hf_reranker = HuggingFaceCrossEncoder(model_name=“BAAI/bge-reranker-base”)
compressor   = CrossEncoderReranker(model=hf_reranker, top_n=4)

base_retriever     = vectorstore.as_retriever(search_kwargs={“k”: 20})
rerank_retriever_lc = ContextualCompressionRetriever(
base_compressor=compressor,

)

1.5 — Mesurer l'impact du reranking

01_benchmark_reranking.py
python 
# Comparer RAG standard vs RAG + Reranking sur un dataset de test
from langsmith import Client
from langsmith.evaluation import evaluate

client = Client()

test_questions = [
{“question”: “Quelle est la politique de remboursement?”, “answer”: “30 jours…”},
{“question”: “Quels modes de paiement sont acceptés?”,    “answer”: “CB, PayPal…”},
]

# Pipeline sans reranking
def rag_standard(inputs):
docs   = vectorstore.similarity_search(inputs[“question”], k=4)
answer = (rag_prompt | llm | StrOutputParser()).invoke({
“context”: fmt_docs(docs), “question”: inputs[“question”]
})
return {“answer”: answer}

# Pipeline avec reranking
def rag_reranked(inputs):
candidates = vectorstore.similarity_search(inputs[“question”], k=20)
docs       = rerank(inputs[“question”], candidates, top_k=4)
answer     = (rag_prompt | llm | StrOutputParser()).invoke({
“context”: fmt_docs(docs), “question”: inputs[“question”]
})
return {“answer”: answer}

# Évaluer les deux et comparer
dataset = client.create_dataset(“reranking-test”)
client.create_examples(
inputs=[{“question”: q[“question”]} for q in test_questions],
outputs=[{“answer”: q[“answer”]} for q in test_questions],
dataset_id=dataset.id
)

res_standard = evaluate(rag_standard, data=“reranking-test”, experiment_prefix=“standard”)
res_reranked  = evaluate(rag_reranked,  data=“reranking-test”, experiment_prefix=“reranked”)
print(“Standard:”, res_standard.to_pandas()[[“feedback.correctness”]].mean())
print(“Reranked:”, res_reranked.to_pandas()[[“feedback.correctness”]].mean())

Quiz — Module 1 : Reranking

3 questions
Q1 · QCM
Quelle est la différence fondamentale entre un Bi-Encoder et un Cross-Encoder ?
ALe Cross-Encoder est plus rapide
BLe Bi-Encoder encode query et doc séparément ; le Cross-Encoder les encode ensemble
CLe Cross-Encoder ne peut pas traiter des paires query/document
DIls produisent exactement les mêmes scores
✓ B. Le Bi-Encoder génère deux vecteurs indépendants → similarité cosinus (rapide, pré-calculable). Le Cross-Encoder encode la paire complète en un seul passage → score de pertinence direct (plus précis, mais calcul à la volée pour chaque paire).
Q2 · QCM
Quel est le pattern optimal pour utiliser le reranking sans exploser la latence ?
AReranker tous les documents de la base
BRécupérer 20-50 candidats avec le retriever, reranker pour garder 3-5
CUtiliser uniquement le Cross-Encoder sans retriever préalable
DReranker 3 candidats récupérés par le retriever
✓ B. "Retrieve large, Rerank small" — le retriever (Bi-Encoder) filtre rapidement les candidats pertinents. Le Cross-Encoder, plus lent mais précis, ne traite que ces candidats. Reranker 20 docs prend ~50ms, reranker 1000 docs prendrait ~2500ms.
Q3 · Ouverte
Expliquez le phénomène "lost in the middle" et comment le reranking l'atténue.
✓ Lost in the middle : les LLMs ont tendance à mieux utiliser les passages en début et fin de contexte, ignorant le milieu (Liu et al., 2023). Sans reranking, le chunk le plus pertinent peut se retrouver en position 3 ou 4 — ignoré par le LLM. Avec reranking, les chunks les plus utiles sont toujours en tête de contexte → le LLM les traite en priorité.

HyDE — Hypothetical Document Embedding

Générer un document hypothétique idéal avant de chercher, pour améliorer drastiquement le rappel sur les questions courtes ou ambiguës.

Intermédiaire

2.1 — Le problème des questions courtes

⚠️

Pourquoi les questions courtes échouent

Une question courte comme "garantie produit?" a un embedding très différent d'un passage long sur la garantie dans un manuel. Le retriever vectoriel cherche la similarité de représentation, pas de sens — et les vecteurs ne se "trouvent" pas facilement.

2.2 — Comment fonctionne HyDE

Question courte
→
LLM génère
un doc hypothétique
→
Embedding
du doc hypothétique
→
Recherche dans
le vectorstore
→
Vrais documents
similaires
💡

L'intuition derrière HyDE

Un document hypothétique sur "la garantie produit" sera dans le même espace d'embedding qu'un vrai document sur ce sujet — même s'il contient des hallucinations. Ce n'est pas la vérité qui compte, c'est la représentation vectorielle. On cherche avec le doc hypothétique, pas avec la réponse hypothétique.

2.3 — Implémentation complète

02_hyde.py
python 
from langchain_ollama import ChatOllama, OllamaEmbeddings
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough, RunnableLambda, RunnableParallel
from langchain_community.vectorstores import Chroma

llm        = ChatOllama(model=“llama3.2”, temperature=0.3)
embeddings = OllamaEmbeddings(model=“nomic-embed-text”)
vectorstore = Chroma(persist_directory=”./chroma_db”, embedding_function=embeddings)

# ── 1. Prompt de génération du document hypothétique ────────────
hyde_prompt = ChatPromptTemplate.from_messages([
(“system”, ””“Tu es un expert qui génère des passages de documentation.
Génère un extrait de documentation technique précis et détaillé
qui répondrait PARFAITEMENT à la question posée.
Écris uniquement le passage, sans introduction ni commentaire.
Longueur: 2-3 paragraphes.”””),
(“human”, “Question: {question}\n\nPassage hypothétique:”),
])

# Chain de génération HyDE
hyde_chain = hyde_prompt | llm | StrOutputParser()

# ── 2. Retriever HyDE ───────────────────────────────────────────
def hyde_retrieve(query: str, k: int = 4) -> list:
””“Retrieval via document hypothétique.”””
# Générer le document hypothétique
hypo_doc = hyde_chain.invoke({“question”: query})

```
# Encoder le document hypothétique (pas la question!)
hypo_embedding = embeddings.embed_query(hypo_doc)

# Chercher les vrais docs similaires au doc hypothétique
docs = vectorstore.similarity_search_by_vector(hypo_embedding, k=k)

return docs
```

# ── 3. HyDE + Reranking (combinaison optimale) ──────────────────
def hyde_rerank_retrieve(query: str, fetch_k: int = 15, top_k: int = 4) -> list:
””“HyDE pour le rappel + Reranking pour la précision.”””
candidates = hyde_retrieve(query, k=fetch_k)
return rerank(query, candidates, top_k=top_k)  # rerank() du module 1

# ── 4. Chain RAG complète avec HyDE ─────────────────────────────
rag_hyde_chain = (
{
“context”:  RunnableLambda(lambda q: fmt_docs(hyde_retrieve(q, k=4))),
“question”: RunnablePassthrough(),
}
| rag_prompt | llm | StrOutputParser()
)

# ── 5. Test comparatif ──────────────────────────────────────────
query = “garantie produit?”

# Standard: embedding de la question courte
std_docs = vectorstore.similarity_search(query, k=4)

# HyDE: embedding du document hypothétique
hypo = hyde_chain.invoke({“question”: query})
print(“Document hypothétique généré:”)
print(hypo[:300], ”…”)

hyde_docs = hyde_retrieve(query, k=4)

print(f”\nDocs standard: {[d.metadata.get(‘source’,’?’) for d in std_docs]}”)
print(f“Docs HyDE:     {[d.metadata.get(‘source’,’?’) for d in hyde_docs]}”)

CRAG — Corrective RAG

Évaluer automatiquement la qualité des documents récupérés et déclencher une recherche corrective si nécessaire.

Avancé
🔄

CRAG — Papier Yan et al. (2024)

CRAG ajoute une étape de validation entre la récupération et la génération. Si les documents sont jugés non pertinents, le système les corrige ou en cherche d'autres (web search). Résultat : -50% d'hallucinations sur les benchmarks.

3.1 — Architecture CRAG

Question
→
Retriever
→
Évaluateur
de pertinence
→
✅ Pertinent → Générer
⚠️ Ambigu → Filtrer + Web
❌ Non pertinent → Web Search

3.2 — Implémentation avec LangGraph

03_crag.py
python 
from langgraph.graph import StateGraph, END
from langchain_ollama import ChatOllama
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.documents import Document
from langchain_community.tools import DuckDuckGoSearchRun
from pydantic import BaseModel, Field
from typing import TypedDict, Annotated, Literal
import operator

llm        = ChatOllama(model=“llama3.2”, temperature=0)
web_search = DuckDuckGoSearchRun()

# ── 1. État CRAG ────────────────────────────────────────────────
class CRAGState(TypedDict):
question:   str
documents:  list[Document]
generation: str
web_search: bool  # True si on a eu besoin d’une recherche web

# ── 2. Évaluateur de pertinence ─────────────────────────────────
class GradeDoc(BaseModel):
score: Literal[“yes”, “no”] = Field(description=”‘yes’ si pertinent, ‘no’ sinon”)

grade_llm = llm.with_structured_output(GradeDoc)

grade_prompt = ChatPromptTemplate.from_messages([
(“system”, ””“Tu es un évaluateur strict de pertinence documentaire.
Réponds ‘yes’ UNIQUEMENT si le document contient des informations
directement utiles pour répondre à la question.
Réponds ‘no’ si le document est hors-sujet, trop général, ou vague.”””),
(“human”, “Question: {question}\n\nDocument:\n{document}\n\nPertinent?”),
])

grade_chain = grade_prompt | grade_llm

# ── 3. Nœuds du graphe CRAG ─────────────────────────────────────
def retrieve(state: CRAGState) -> dict:
””“Récupère les documents depuis le vectorstore.”””
docs = vectorstore.similarity_search(state[“question”], k=4)
return {“documents”: docs}

def grade_documents(state: CRAGState) -> dict:
””“Évalue chaque document — ne garde que les pertinents.”””
question  = state[“question”]
documents = state[“documents”]

```
relevant_docs = []
need_web      = False

for doc in documents:
    grade = grade_chain.invoke({
        "question": question,
        "document": doc.page_content
    })
    if grade.score == "yes":
        relevant_docs.append(doc)
    else:
        need_web = True  # Au moins un doc non pertinent

# Si aucun doc pertinent → on a besoin du web
if not relevant_docs:
    need_web = True

return {"documents": relevant_docs, "web_search": need_web}
```

def web_search_node(state: CRAGState) -> dict:
””“Effectue une recherche web et ajoute les résultats aux documents.”””
question   = state[“question”]
web_result = web_search.run(question)

```
# Convertir le résultat web en Document
web_doc = Document(
    page_content=web_result,
    metadata={"source": "web_search", "type": "web"}
)

# Ajouter aux documents existants (s'il y en a)
existing = state["documents"]
return {"documents": existing + [web_doc]}
```

def generate(state: CRAGState) -> dict:
””“Génère la réponse finale.”””
question  = state[“question”]
documents = state[“documents”]

```
if not documents:
    return {"generation": "Je n'ai pas trouvé d'informations pertinentes."}

context = fmt_docs(documents)
answer  = (rag_prompt | llm | StrOutputParser()).invoke({
    "context": context, "question": question
})
return {"generation": answer}
```

# ── 4. Condition de routage ──────────────────────────────────────
def route_after_grading(state: CRAGState) -> Literal[“web_search”, “generate”]:
””“Décide si on fait une recherche web ou on génère directement.”””
if state[“web_search”]:
return “web_search”
return “generate”

# ── 5. Construire le graphe CRAG ─────────────────────────────────
crag_builder = StateGraph(CRAGState)
crag_builder.add_node(“retrieve”,        retrieve)
crag_builder.add_node(“grade_documents”, grade_documents)
crag_builder.add_node(“web_search”,      web_search_node)
crag_builder.add_node(“generate”,        generate)

crag_builder.set_entry_point(“retrieve”)
crag_builder.add_edge(“retrieve”, “grade_documents”)
crag_builder.add_conditional_edges(“grade_documents”, route_after_grading)
crag_builder.add_edge(“web_search”, “generate”)
crag_builder.add_edge(“generate”,   END)

crag_graph = crag_builder.compile()

# ── 6. Utilisation ──────────────────────────────────────────────
result = crag_graph.invoke({
“question”: “Quel est le dernier modèle LLaMA d’OpenAI?”,
“documents”: [], “generation”: ””, “web_search”: False
})
print(result[“generation”])
print(f“Recherche web utilisée: {result[‘web_search’]}”)

3.3 — CRAG amélioré — Réécriture de requête

# Avant la recherche web, réécrire la question pour le moteur de recherche
def rewrite_query(state: CRAGState) -> dict:
    """Réécrit la question pour optimiser la recherche web."""
    rewrite_prompt = ChatPromptTemplate.from_messages([
        ("system", """Tu transformes des questions en requêtes de recherche web optimales.
        Retourne UNIQUEMENT la requête, sans explication."""),
        ("human", "Question: {question}\nRequête web optimisée:"),
    ])
    query = (rewrite_prompt | llm | StrOutputParser()).invoke({
        "question": state["question"]
    })
    web_result = web_search.run(query.strip())
    web_doc    = Document(page_content=web_result,
                           metadata={"source": "web", "query": query})
    return {"documents": state["documents"] + [web_doc]}

Quiz — Module 3 : CRAG

2 questions
Q1 · QCM
Dans CRAG, que se passe-t-il si certains documents récupérés sont pertinents et d'autres ne le sont pas ?
AOn abandonne tous les documents et on fait uniquement une recherche web
BOn garde les pertinents et on complète avec une recherche web
COn utilise tous les documents, pertinents ou non
DOn régénère une nouvelle requête de retrieval
✓ B. L'implémentation optimale de CRAG filtre les documents non pertinents, garde les bons, puis complète avec une recherche web. On met need_web = True dès qu'au moins un document est rejeté, mais on cumule les sources pertinentes.
Q2 · Ouverte
Pourquoi est-il important de réécrire la requête avant la recherche web dans CRAG ?
✓ La question originale est formulée pour un humain ou un vectorstore. Une requête web efficace est différente : plus courte, avec des mots-clés précis, sans mots de liaison. Par exemple, "Quels sont les avantages du RAG?" devient "RAG advantages limitations LLM". La réécriture améliore significativement la qualité des résultats web.

Self-RAG — Retrieval par Réflexion

Le LLM décide lui-même quand chercher, évalue ce qu'il trouve et critique sa propre réponse avant de la livrer.

Avancé
🧠

Self-RAG — Asai et al. (2023)

Self-RAG introduit des tokens de réflexion spéciaux que le LLM génère pour contrôler son propre processus : [Retrieve], [Relevant], [IsSupported], [IsUseful]. Dans notre implémentation pratique avec LangGraph, on simule ce comportement avec des nœuds dédiés.

4.1 — Les 4 décisions Self-RAG

Token / DécisionQuestion poséeValeurs possibles
Retrieve?Ai-je besoin de chercher des documents?Yes / No
IsRelevantCe document est-il utile pour ma réponse?Relevant / Irrelevant
IsSupportedMa réponse est-elle soutenue par les docs?Fully / Partially / Not
IsUsefulMa réponse est-elle utile pour l'utilisateur?Score 1-5

4.2 — Implémentation Self-RAG avec LangGraph

04_self_rag.py
python 
from langgraph.graph import StateGraph, END
from langchain_ollama import ChatOllama
from langchain_core.prompts import ChatPromptTemplate
from pydantic import BaseModel, Field
from typing import TypedDict, Annotated, Literal, Optional
import operator

llm = ChatOllama(model=“llama3.2”, temperature=0)

# ── 1. État Self-RAG ────────────────────────────────────────────
class SelfRAGState(TypedDict):
question:     str
documents:    list
generation:   str
iterations:   int          # compteur d’itérations
is_supported: str          # “yes” / “partial” / “no”

# ── 2. Schémas Pydantic pour les décisions ─────────────────────
class NeedRetrieval(BaseModel):
need: Literal[“yes”, “no”] = Field(description=”‘yes’ si besoin de docs, ‘no’ sinon”)

class Hallucination(BaseModel):
supported: Literal[“yes”, “partial”, “no”] = Field(
description=”‘yes’=bien supporté, ‘partial’=partiellement, ‘no’=hallucination”
)

class AnswerQuality(BaseModel):
useful: Literal[“yes”, “no”] = Field(description=”‘yes’ si la réponse est utile”)

# ── 3. Chains de décision ───────────────────────────────────────
retrieval_chain = (
ChatPromptTemplate.from_messages([
(“system”, ””“Décide si tu as besoin de documents pour répondre.
Réponds ‘no’ si: question de politesse, calcul, question factuelle simple.
Réponds ‘yes’ si: question sur des données spécifiques, procédures, faits précis.”””),
(“human”, “Question: {question}”),
])
| llm.with_structured_output(NeedRetrieval)
)

hallucination_chain = (
ChatPromptTemplate.from_messages([
(“system”, ””“Évalue si la réponse est bien soutenue par les documents fournis.
‘yes’ = chaque affirmation est prouvée par les docs.
‘partial’ = certaines affirmations ne sont pas prouvées.
‘no’ = la réponse contient des informations absentes des docs (hallucination).”””),
(“human”, “Documents:\n{documents}\n\nRéponse générée:\n{generation}”),
])
| llm.with_structured_output(Hallucination)
)

quality_chain = (
ChatPromptTemplate.from_messages([
(“system”, “Évalue si la réponse répond bien à la question posée. ‘yes’ ou ‘no’.”),
(“human”, “Question: {question}\nRéponse: {generation}”),
])
| llm.with_structured_output(AnswerQuality)
)

# ── 4. Nœuds du graphe ──────────────────────────────────────────
def decide_retrieval(state: SelfRAGState) -> dict:
””“Décide si on a besoin de retrieval.”””
decision = retrieval_chain.invoke({“question”: state[“question”]})
return {“need_retrieval”: decision.need}

def retrieve_docs(state: SelfRAGState) -> dict:
””“Récupère les documents.”””
docs = vectorstore.similarity_search(state[“question”], k=4)
return {“documents”: docs}

def generate_answer(state: SelfRAGState) -> dict:
””“Génère une réponse (avec ou sans docs).”””
question = state[“question”]
docs     = state.get(“documents”, [])

```
if docs:
    context = fmt_docs(docs)
    answer  = (rag_prompt | llm | StrOutputParser()).invoke({
        "context": context, "question": question
    })
else:
    answer = llm.invoke(question).content

return {"generation": answer, "iterations": state.get("iterations", 0) + 1}
```

def check_hallucination(state: SelfRAGState) -> dict:
””“Vérifie si la réponse est bien ancrée dans les documents.”””
docs = state.get(“documents”, [])
if not docs:
return {“is_supported”: “yes”}  # Pas de docs = pas de vérification

```
result = hallucination_chain.invoke({
    "documents": fmt_docs(docs),
    "generation": state["generation"]
})
return {"is_supported": result.supported}
```

# ── 5. Conditions de routage ─────────────────────────────────────
def route_initial(state: SelfRAGState) -> str:
return “retrieve” if state.get(“need_retrieval”) == “yes” else “generate”

def route_after_hallucination(state: SelfRAGState) -> str:
supported  = state[“is_supported”]
iterations = state.get(“iterations”, 0)

```
if supported == "yes":
    # Vérifier la qualité avant de finir
    quality = quality_chain.invoke({
        "question": state["question"],
        "generation": state["generation"]
    })
    return "end" if quality.useful == "yes" else "regenerate"

elif supported == "partial" and iterations < 2:
    return "retrieve"  # Chercher plus de docs

else:
    return "regenerate" if iterations < 3 else "end"
```

# ── 6. Construire le graphe ──────────────────────────────────────
sg = StateGraph(SelfRAGState)
for name, fn in [(“decide”, decide_retrieval), (“retrieve”, retrieve_docs),
(“generate”, generate_answer), (“check”, check_hallucination)]:
sg.add_node(name, fn)

sg.set_entry_point(“decide”)
sg.add_conditional_edges(“decide”,  route_initial,
{“retrieve”: “retrieve”, “generate”: “generate”})
sg.add_edge(“retrieve”, “generate”)
sg.add_edge(“generate”, “check”)
sg.add_conditional_edges(“check”, route_after_hallucination, {
“end”:         END,
“retrieve”:   “retrieve”,   # Chercher plus
“regenerate”: “generate”,   # Régénérer
})

self_rag = sg.compile()

# ── 7. Test ──────────────────────────────────────────────────────
result = self_rag.invoke({
“question”: “Quelle est la politique de retour produit?”,
“documents”: [], “generation”: ””, “iterations”: 0, “is_supported”: ””
})
print(result[“generation”])
print(f“Itérations: {result[‘iterations’]}”)
💡

Conseil pratique — Limiter les itérations

Self-RAG peut boucler si le LLM est trop strict. Définissez toujours un max_iterations=3 et un fallback vers une réponse honnête ("Je n'ai pas trouvé d'information suffisante") plutôt qu'une boucle infinie.

Module 5 · Expert

GraphRAG — Graphe de Connaissances

Enrichir le RAG avec un graphe de connaissances pour capturer les relations entre entités et répondre à des questions multi-sauts.

Expert

5.1 — Pourquoi le RAG vectoriel a ses limites

⚠️

Questions multi-sauts — le talon d'Achille du RAG

Exemple : "Quelle est la ville natale du CEO de l'entreprise qui a acquis Twitter?"
Un RAG vectoriel cherche un chunk unique. Mais la réponse nécessite 3 relations : Twitter → acquéreur → CEO → ville natale. Un graphe de connaissances navigue ces relations naturellement.

5.2 — Architecture GraphRAG

RAG Vectoriel seul
  • Cherche des chunks similaires
  • Pas de navigation entre entités
  • Questions multi-sauts impossibles
  • Pas de contexte relationnel
GraphRAG
  • Extrait entités + relations des chunks
  • Construit un graphe de connaissances
  • Navigue les relations (multi-sauts)
  • Combine vectorstore + graphe

class Triplet(BaseModel):
sujet:     str = Field(description=“Entité sujet”)
relation:  str = Field(description=“Type de relation”)
objet:     str = Field(description=“Entité objet”)

extract_prompt = ChatPromptTemplate.from_messages([
(“system”, ””“Extrais tous les triplets (sujet, relation, objet) du texte.
Relations utiles: est_CEO_de, a_acquis, est_situé_à, travaille_pour,
est_fondé_par, a_produit, est_concurrent_de, est_filiale_de, etc.
Normalise les noms propres (majuscules). Sois exhaustif.”””),
(“human”, “Texte:\n{text}\n\nTriplets:”),
])

extract_chain = extract_prompt | llm.with_structured_output(Triplets)

def extract_triplets(text: str) -> list[Triplet]:
try:
result = extract_chain.invoke({“text”: text[:2000]})
return result.triplets
except:
return []

# ── 3. Construire le graphe de connaissances ────────────────────
def build_knowledge_graph(documents: list) -> nx.DiGraph:
””“Construit un graphe dirigé à partir des documents.”””
G = nx.DiGraph()

```
for doc in documents:
    triplets = extract_triplets(doc.page_content)
    for t in triplets:
        G.add_node(t.sujet, type="entity")
        G.add_node(t.objet, type="entity")
        G.add_edge(
            t.sujet, t.objet,
            relation=t.relation,
            source=doc.metadata.get("source", "?")
        )

print(f"Graphe: {G.number_of_nodes()} nœuds, {G.number_of_edges()} relations")
return G
```

# ── 4. Requête sur le graphe ────────────────────────────────────
def query_graph(G: nx.DiGraph, entity: str, depth: int = 2) -> str:
””“Récupère le sous-graphe autour d’une entité.”””
if entity not in G.nodes:
# Cherche une entité similaire (insensible à la casse)
entity = next(
(n for n in G.nodes if entity.lower() in n.lower()),
None
)
if not entity:
return f”Entité non trouvée dans le graphe.”

```
# Sous-graphe à N degrés de profondeur
sub_nodes = nx.ego_graph(G, entity, radius=depth).nodes
subgraph  = G.subgraph(sub_nodes)

# Formater les relations comme texte
triplets = []
for u, v, data in subgraph.edges(data=True):
    triplets.append(f"{u} --[{data['relation']}]--> {v}")

return "\n".join(triplets) if triplets else "Aucune relation trouvée."
```

# ── 5. Extraire les entités d’une question ──────────────────────
entity_prompt = ChatPromptTemplate.from_messages([
(“system”, “Extrais les entités nommées (personnes, entreprises, produits, lieux) de la question. Retourne une liste Python.”),
(“human”, “Question: {question}\nEntités (liste Python):”),
])

def extract_entities(question: str) -> list[str]:
result = (entity_prompt | llm | StrOutputParser()).invoke({“question”: question})
try:
return eval(result.strip())  # [“Entité1”, “Entité2”]
except:
return []

# ── 6. GraphRAG Chain complète ───────────────────────────────────
def graphrag_query(question: str, G: nx.DiGraph, vectorstore) -> str:
””“Combine contexte vectoriel + contexte graphe.”””

```
# Contexte vectoriel classique
vector_docs    = vectorstore.similarity_search(question, k=3)
vector_context = fmt_docs(vector_docs)

# Contexte graphe (relations entre entités)
entities      = extract_entities(question)
graph_context = "\n\n".join([
    f"Relations pour '{e}':\n{query_graph(G, e)}"
    for e in entities
]) if entities else ""

# Prompt enrichi avec les deux contextes
graphrag_prompt = ChatPromptTemplate.from_template("""
Réponds à la question en utilisant:
1. Les documents récupérés
2. Le graphe de relations entre entités

Documents:
{vector_context}

Relations (graphe de connaissances):
{graph_context}

Question: {question}
Réponse:""")

answer = (graphrag_prompt | llm | StrOutputParser()).invoke({
    "vector_context": vector_context,
    "graph_context":  graph_context or "Aucune relation trouvée.",
    "question":       question
})
return answer
```

# ── 7. Usage complet ────────────────────────────────────────────

Adaptive RAG — Routage Intelligent

Router dynamiquement chaque question vers la stratégie RAG optimale — simple, avancé ou sans RAG — pour le meilleur équilibre vitesse/qualité.

Expert

6.1 — Le principe de l'Adaptive RAG

🎯

Pareto — Ne pas appliquer la même stratégie à toutes les questions

"Bonjour, comment vas-tu?" ne nécessite pas de RAG. "Explique-moi le fonctionnement du modèle GPT-4" ne nécessite pas vos docs internes. "Quelle est la politique de retour?" nécessite un RAG simple. "Compare nos ventes Q3 avec nos concurrents" nécessite un RAG complexe + web. Router intelligemment = ×3 sur la performance.

6.2 — Implémentation Adaptive RAG

06_adaptive_rag.py

from langgraph.graph import StateGraph, END
from pydantic import BaseModel, Field
from typing import TypedDict, Literal

llm = ChatOllama(model=“llama3.2”, temperature=0)

# ── 1. État ─────────────────────────────────────────────────────
class AdaptiveState(TypedDict):
question:   str
strategy:   str   # “none” / “simple” / “advanced” / “web”
documents:  list
generation: str

# ── 2. Classificateur de stratégie ──────────────────────────────
class Strategy(BaseModel):
strategy: Literal[“none”, “simple”, “advanced”, “web”] = Field(
description=””“Stratégie à utiliser:
- none: question conversationnelle, salutation, calcul simple
- simple: question sur les docs internes, réponse dans 1-2 chunks
- advanced: question complexe, multi-docs, analyse comparative
- web: question sur l’actualité, événements récents”””
)
reason: str = Field(description=“Raison du choix”)

router_chain = (
ChatPromptTemplate.from_messages([
(“system”, ””“Tu es un routeur de questions pour un système RAG.
Classe chaque question dans la stratégie adaptée.
Base de connaissance: documentation produit et procédures internes.”””),
(“human”, “Question: {question}”),
])
| llm.with_structured_output(Strategy)
)

# ── 3. Nœuds spécialisés ────────────────────────────────────────
def classify(state: AdaptiveState) -> dict:
result = router_chain.invoke({“question”: state[“question”]})
print(f“Stratégie: {result.strategy} — {result.reason}”)
return {“strategy”: result.strategy}

def generate_direct(state: AdaptiveState) -> dict:
””“Répond sans RAG — pour les questions simples.”””
answer = llm.invoke(state[“question”]).content
return {“generation”: answer, “documents”: []}

def rag_simple(state: AdaptiveState) -> dict:
””“RAG standard avec 4 chunks.”””
docs   = vectorstore.similarity_search(state[“question”], k=4)
answer = (rag_prompt | llm | StrOutputParser()).invoke({
“context”: fmt_docs(docs), “question”: state[“question”]
})
return {“generation”: answer, “documents”: docs}

def rag_advanced(state: AdaptiveState) -> dict:
””“RAG avancé avec HyDE + Reranking + CRAG.”””
question = state[“question”]

```
# HyDE pour améliorer le rappel
candidates = hyde_retrieve(question, k=15)

# Reranking pour la précision
docs = rerank(question, candidates, top_k=5)

# Vérification CRAG rapide
relevant = [
    d for d in docs
    if grade_chain.invoke({"question": question, "document": d.page_content}).score == "yes"
]
if not relevant:
    relevant = docs  # fallback si aucun n'est jugé parfait

answer = (rag_prompt | llm | StrOutputParser()).invoke({
    "context": fmt_docs(relevant), "question": question
})
return {"generation": answer, "documents": relevant}
```

def rag_with_web(state: AdaptiveState) -> dict:
””“RAG + recherche web pour les questions d’actualité.”””
question = state[“question”]

```
# Docs internes
int_docs = vectorstore.similarity_search(question, k=2)

# Recherche web
web_result = web_search.run(question)
web_doc    = Document(page_content=web_result, metadata={"source": "web"})

all_docs = int_docs + [web_doc]
answer   = (rag_prompt | llm | StrOutputParser()).invoke({
    "context": fmt_docs(all_docs), "question": question
})
return {"generation": answer, "documents": all_docs}
```

# ── 4. Routage ───────────────────────────────────────────────────
def route(state: AdaptiveState) -> str:
return {
“none”:     “direct”,
“simple”:   “simple”,
“advanced”: “advanced”,
“web”:      “web”,
}.get(state[“strategy”], “simple”)

# ── 5. Graphe Adaptive RAG ───────────────────────────────────────
ag = StateGraph(AdaptiveState)
ag.add_node(“classify”,  classify)
ag.add_node(“direct”,    generate_direct)
ag.add_node(“simple”,    rag_simple)
ag.add_node(“advanced”,  rag_advanced)
ag.add_node(“web”,       rag_with_web)

ag.set_entry_point(“classify”)
ag.add_conditional_edges(“classify”, route,
{“direct”:“direct”,“simple”:“simple”,“advanced”:“advanced”,“web”:“web”})
for node in [“direct”,“simple”,“advanced”,“web”]:
ag.add_edge(node, END)

adaptive_graph = ag.compile()

# ── 6. Tests ─────────────────────────────────────────────────────
tests = [
“Bonjour, comment vas-tu?”,          # → none
“Quelle est la durée de garantie?”,  # → simple
“Compare nos produits avec ceux de nos concurrents”,  # → advanced
“Quelles sont les dernières nouvelles sur les LLMs?”,  # → web
]
for q in tests:
result = adaptive_graph.invoke({
“question”: q, “strategy”: ””, “documents”: [], “generation”: ””
})
print(f”[{result[‘strategy’]}] {q[:40]}…”)
print(f“→ {result[‘generation’][:100]}…\n”)

# Pipeline RAG expert — toutes les techniques combinées
from langsmith import traceable
import time

class ExpertRAGPipeline:
””“Pipeline RAG production combinant toutes les techniques.”””

```
def __init__(self, vectorstore, graph=None):
    self.vs       = vectorstore
    self.graph    = graph
    self.reranker = CrossEncoder("BAAI/bge-reranker-base")
    self.llm      = ChatOllama(model="llama3.2", temperature=0)
    
    # Retriever hybride de base
    bm25     = BM25Retriever.from_documents(all_docs); bm25.k = 10
    vec_r    = vectorstore.as_retriever(search_kwargs={"k": 10})
    self.base_retriever = EnsembleRetriever(
        retrievers=[bm25, vec_r], weights=[0.4, 0.6]
    )

@traceable(name="expert_rag")
def query(self, question: str, strategy: str = "auto") -> dict:
    """Exécute le pipeline complet."""
    t0 = time.time()
    
    if strategy == "auto":
        strategy = self._classify(question)
    
    docs   = self._retrieve(question, strategy)
    answer = self._generate(question, docs, strategy)
    
    return {
        "answer":   answer,
        "sources":  [d.metadata.get("source") for d in docs],
        "strategy": strategy,
        "latency":  round(time.time() - t0, 2),
    }

def _classify(self, q: str) -> str:
    result = router_chain.invoke({"question": q})
    return result.strategy

def _retrieve(self, q: str, strategy: str) -> list:
    if strategy == "none":
        return []
    
    if strategy == "simple":
        # Hybride BM25+vectoriel + reranking
        candidates = self.base_retriever.get_relevant_documents(q)
        return rerank(q, candidates, top_k=4)
    
    elif strategy == "advanced":
        # HyDE pour le rappel + hybride + reranking lourd
        hyde_docs  = hyde_retrieve(q, k=10)
        base_docs  = self.base_retriever.get_relevant_documents(q)
        candidates = list({d.page_content: d for d in hyde_docs + base_docs}.values())
        
        # Filtrage CRAG
        relevant = [
            d for d in candidates
            if grade_chain.invoke({"question": q, "document": d.page_content}).score == "yes"
        ] or candidates  # fallback
        
        return rerank(q, relevant, top_k=5)
    
    elif strategy == "web":
        int_docs   = self.base_retriever.get_relevant_documents(q)[:2]
        web_result = web_search.run(q)
        web_doc    = Document(page_content=web_result, metadata={"source":"web"})
        return int_docs + [web_doc]
    
    return self.base_retriever.get_relevant_documents(q)[:4]

def _generate(self, q: str, docs: list, strategy: str) -> str:
    if not docs:
        return self.llm.invoke(q).content
    
    # Enrichir avec le graphe si disponible et stratégie avancée
    graph_ctx = ""
    if self.graph and strategy == "advanced":
        entities  = extract_entities(q)
        graph_ctx = "\n".join(query_graph(self.graph, e) for e in entities)
    
    context = fmt_docs(docs)
    if graph_ctx:
        context += f"\n\nRelations graphe:\n{graph_ctx}"
    
    return (rag_prompt | self.llm | StrOutputParser()).invoke({
        "context": context, "question": q
    })
```

4 questions
Q1 · QCM
Dans quel ordre appliquer Reranking et HyDE pour obtenir la meilleure précision ?
AReranking d'abord, puis HyDE
BHyDE pour récupérer les candidats, Reranking pour les classer
CReranking et HyDE en parallèle
DL'ordre n'a aucune importance
✓ B. HyDE améliore le rappel (trouver les bons docs), Reranking améliore la précision (les mettre dans le bon ordre). L'ordre logique : HyDE → large pool de candidats → Reranking → top-k précis. Inverser n'aurait aucun sens.
Q2 · QCM
Quelle technique est la plus efficace pour les questions nécessitant de traverser plusieurs relations entre entités ?
ASelf-RAG (boucle de réflexion)
BCRAG (validation des documents)
CGraphRAG (navigation du graphe de connaissances)
DHyDE (document hypothétique)
✓ C. Les questions multi-sauts (A→B→C) nécessitent de naviguer des relations. Un vectorstore cherche des passages similaires mais ne "suit" pas les relations. Le graphe de connaissances est structurellement fait pour cela : MATCH (a)-[:REL1]->(b)-[:REL2]->(c).
Q3 · Architecture
Pour un chatbot de support client avec 50 000 documents PDF, une latence max de 3 secondes et 99% de précision requise, quelle architecture choisiriez-vous et pourquoi ?
✓ Architecture recommandée :

Indexation : Chroma ou Qdrant (production) avec embeddings nomic-embed-text. Chunking 1000/200 chars. Métadonnées riches (catégorie, date, produit).

Retrieval : EnsembleRetriever BM25+vectoriel (fetch_k=20) → Cross-Encoder BGE-Reranker (top_k=5). Latence ~400ms.

Pipeline : Adaptive RAG pour le routage → CRAG pour la validation (web search si docs insuffisants). Self-RAG désactivé en prod (trop lent). GraphRAG uniquement pour les questions de comparaison produits.

Optimisations : Cache Redis (requêtes fréquentes), streaming côté client (UX), LangSmith pour monitoring continu des scores faithfulness.

Pourquoi pas Self-RAG ? La boucle de réflexion peut prendre 5-15s. Pour la contrainte de 3s, CRAG + Reranking offre 95% de la qualité à 20% de la latence.
Q4 · Ouverte
Comment mesureriez-vous l'amélioration apportée par chaque technique ajoutée à votre pipeline RAG ?
✓ Méthodologie A/B rigoureuse :

1. Golden Dataset : 50+ questions avec réponses validées manuellement par des experts métier.
2. Métriques : Faithfulness (réponse ancrée dans les docs), Correctness (vs réponse attendue), Latency P50/P95.
3. Baseline : RAG simple (vectorstore + LLM), mesurer les métriques.
4. +Reranking : mesurer le delta. +Reranking typiquement +20-40% correctness.
5. +HyDE : mesurer le delta supplémentaire. Utile surtout sur questions courtes.
6. +CRAG : mesurer réduction hallucinations (-50% attendu).
7. Décision : ajouter une technique seulement si gain > coût de latence ajoutée.

J'utilise un EnsembleRetriever (BM25 + vectoriel) comme base

J'évalue la pertinence des documents avec CRAG
J'ai un golden dataset d'au moins 30 questions critiques
